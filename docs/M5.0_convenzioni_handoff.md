## Cosa è stato costruito
- **DB**: tabella `convenzioni` (RLS on, deny-by-default per anon) + vista pubblica `convenzioni_pubbliche` (solo `stato='attiva'`, MAI referente) + tabella `convenzioni_rate_limit` + funzione `convenzioni_rl_hit(text,int)`. Migrazione: `supabase/migrations/20260709170000_convenzioni.sql` (applicata via Management API).
- **Edge function** `convenzioni-proposta` (deployata, `--no-verify-jwt`): una sola function, due compiti:
  - `POST /` (dal form): valida → INSERT `proposta` → email al Direttivo con 2 link HMAC → email di cortesia al referente. Ordine INSERT→email (lezione A2).
  - `GET/POST .../azione/{approva|rifiuta}/{id}/{exp}/{t}`: GET mostra conferma, POST esegue l'UPDATE idempotente (solo se ancora `proposta`). Token HMAC `ADMIN_ACTION_SECRET`, scope distinti, TTL 30gg.
- **Pagine**: `/convenzioni` (SSR, `prerender=false`, legge la vista; griglia nascosta se 0 attive) e `/convenzioni/schema-tipo` (fac-simile). Riga "Convenzioni" aggiunta in `/tesseramento`.

## Link HMAC nelle email — nota importante
I parametri stanno nel **path**, non in query string: un URL `?id=1e...` viene **corrotto** dall'encoding quoted-printable delle email (`=` + 2 cifre esadecimali → byte). Verificato dal vivo. Struttura corretta:
`…/convenzioni-proposta/azione/approva/{uuid}/{exp}/{token-hex}`
⚠️ Questo bug affligge probabilmente anche i link di `scheda-domanda` (funzione protetta, non toccata): da valutare in un fix dedicato.
